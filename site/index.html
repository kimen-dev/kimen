Why it exists

A guarded catalog, by design

The planned renderer will accept only catalog components and declared actions, reject unknown props and expose no code-execution path from spec data. That guardrail will be tested as a security boundary before it is presented as shipped.

Any brand, one attribute

Zero hardcoded visual values. Components read tokens; brands reassign the semantic layer. Mars is the default theme, never a requirement — a Material 3 reference theme re-skins every component in CI on every commit.

Done means gates exit 0

Real-browser tests, zero axe violations, packaging checks, scenario-to-test traceability and per-component size budgets. Mutation gating is part of the current integrity-hardening work. No self-assessment — a script proves each completed gate.

Legible to agents, by contract

The component contract is the single source of truth. Everything an agent needs is generated from it and shipped with the package — an API a capable agent cannot wire correctly from its description alone is treated as a defect.

  • custom-elements.json — machine-readable manifest of every prop, event, slot and part
  • llms.txt — the library, summarized for language models
  • when-to-use / when-NOT-to-use metadata on every component
  • planned: a Zod-validated catalog schema for runtime GenUI (A2UI, MCP Apps, AG-UI, json-render)
