# Kimen component workshop

**Kimen** is a neutral, multi-protocol GenUI catalog in web components: an
agent emits a schema-constrained UI spec, a guardrail validates it against a
closed catalog, and standards-first `<ki-*>` elements render it — in any
host, whatever the framework, under any brand.

This Storybook is the **component workshop**: every published component with
its props as live controls, its docs generated from the component contract
(the TypeScript types + JSDoc are the single source of truth), and an
accessibility panel wired to axe-core.

## How to read a component page

- **Playground** — every prop exposed as a control; slots shown as text.
- **Named stories** — one story per API axis (variants, tones, sizes…), so
  each design decision is visible in isolation.
- **Docs tab** — autogenerated from the manifest: props, events, slots,
  parts and CSS custom properties. If something is undocumented there, that
  is a build failure, not a docs gap.

## Theming in the toolbar

The **Scheme** toolbar switches light / dark / auto — it drives the same
`data-ki-color-scheme` attribute a consuming page would use. See
*Docs → Tokens & theming* for the full model.

## Where things live

- Landing & live re-theming demo: [kimen-dev.github.io/kimen](https://kimen-dev.github.io/kimen/)

- Status: **pre-v1** — components land one by one through an approved-spec
  pipeline with deterministic quality gates (real-browser tests, axe zero
  violations, mutation-score gates, bundle budgets).

## For agents

Kimen treats machine legibility as a deliverable: components ship
when-to-use / when-NOT-to-use metadata, a generated manifest, and (as the
catalog phase lands) a Zod-validated schema of what agents may emit. An API
a capable agent cannot wire correctly from its description alone is a
defect — file it as one.
